# HTTP Server Architecture

Status: HTTP-SERVER-01 internal alpha contract.

## Scope

This document covers inbound HTTP/1.1 server runtime behavior. It does not cover TLS,
HTTP/2, HTTP/3, WebSockets, static files, reverse proxy behavior, outbound HTTP client
behavior, public alpha documentation, or production-edge claims.

## Layers

The server stack is layered so future protocols can bind the same app/framework shape:

- parser: bounded HTTP/1 request-head and body validation;
- backend: listener, connection, request, body-reader, cancellation, timeout, and shutdown
  lifecycle state;
- transport: libuv-owned localhost TCP handles, bounded read/write buffers, timers, and
  sequential keep-alive;
- dispatch: Plan route table, route/query/body policy, request-context materialization,
  and engine call boundary;
- engine bridge: V8 request-context object creation and result descriptor conversion.

Socket, libuv, V8, and native handle internals do not cross into app-facing objects.

## Parser And Header Policy

The complete-buffer parser accepts HTTP/1.0 and HTTP/1.1 request heads with origin-form
targets. HTTP/1.1 requires exactly one non-empty `Host` header. HTTP/1.0 may omit `Host`,
but any present Host header must be singular and non-empty. Duplicate `Content-Length`,
duplicate `Transfer-Encoding`, and `Content-Length` plus `Transfer-Encoding` conflicts are
rejected before dispatch.

The parser enforces bounded request-line, target, header count, header-name, header-value,
total-header-byte, and body limits. Request bodies preserve binary bytes; app-facing body
helpers decide whether those bytes are text, JSON, or octet-stream data.

## Request Body

Supported request body forms are:

- empty body;
- bounded `Content-Length` body;
- bounded chunked body decoded by the transport before parser/body policy handoff.

Supported app-facing body media are `application/json`, `text/plain`, and
`application/octet-stream`. JSON is validated before handler entry. Unsupported media
types, unsupported transfer codings, malformed chunk framing, trailers, body limit
violations, short bodies, timeouts, cancellation, shutdown, and disconnects produce stable
diagnostics. There is no unbounded buffering and no public request streaming API in this
contract.

## Response Writer

Fixed native responses serialize status, safe headers, content type, content length, and
body bytes. Scoped native/runtime streaming responses use chunked transfer encoding and
validate caller-provided headers before headers are started. Managed headers such as
`Connection`, `Content-Type`, `Content-Length`, and `Transfer-Encoding` remain
transport-owned in the streaming path.

Public mutable response writers, `onStarting`, `onCompleted`, public response streaming
helpers, and public streaming examples are deferred.

## Connection And Lifecycle

The transport implements sequential HTTP/1.1 keep-alive. A connection can process a new
request only after the previous response write completes and parser, request, body reader,
response, timeout, and lifecycle state are reset. Pipelined bytes are rejected
deterministically because queued response ownership and queued cancellation semantics are
not implemented.

Header read, body read, total request, write, idle keep-alive, disconnect, shutdown, and
force-close paths release backend request/connection admission exactly once. Late write or
close completions are cleanup-only. Timeout, external cancellation, client disconnect, and
shutdown use distinct diagnostics where surfaced.

## Request Context

`SlHttpRequestContext` is an adapter input, not the public protocol contract. The V8 bridge
materializes a protocol-abstract object with request, route, query, connection, signal, and
deadline fields. Request IDs and connection IDs are generated by the native lifecycle and
exposed as strings when available. Synthetic dispatch paths that do not own a transport
lifecycle expose `"0"` IDs.

The context does not expose HTTP parser state, socket state, libuv handles, V8 types,
native pointers, or OS handles.

## Config

`sloppy run` consumes these server config metadata keys when emitted by the Plan/config
pipeline:

- `Sloppy:Server:Host`;
- `Sloppy:Server:Port`;
- `Sloppy:Server:MaxConnections`;
- `Sloppy:Server:MaxRequestBodyBytes`;
- `Sloppy:Server:RequestTimeoutMs`;
- `Sloppy:Server:KeepAliveEnabled`;
- `Sloppy:Server:KeepAliveIdleTimeoutMs`;
- `Sloppy:Server:MaxRequestsPerConnection`.

Malformed, zero, unsupported, or range-overflowing numeric values fail closed before the
server starts.

## Deferred Contract

The following remain explicit follow-up work rather than hidden behavior:

- route-level body/header/timeout limits from route metadata;
- trusted proxy configuration and forwarded header interpretation;
- request ID adoption from trusted headers;
- access-event and counter output;
- public response lifecycle callbacks and stream helpers;
- production graceful drain and edge tuning.
